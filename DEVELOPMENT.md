# LynxKit 开发流程规范

> 本文档为 LynxKit 项目的强制开发规范。每次接收任务/需求时必须严格遵守，不得跳过任何环节。
>
> **核心原则**：能少写就不多写；能用简单方案就不做复杂；能共用组件/代码就绝不重复写；找到的 bug 必须彻底修复，不得搁置。

## 1. 需求澄清与不确定点确认（任务全程）

**触发时机**：任务启动前 + 任务进行中遇到任何不确定点时。

### 1.1 任务启动前的需求澄清
- 收到任务后，先通读任务描述与上下文（含 memory、topics、相关文件）。
- 如存在以下任一情况，**必须**通过对话向用户确认，完全澄清后才能开始实现：
  - 需求描述存在歧义或多种合理解读
  - 涉及的技术选型、依赖方向、对外契约不明确
  - 任务范围边界模糊（不知道做不做、改不改）
  - 与既有架构、约束、已沉淀的 project_memory 存在潜在冲突
  - 验收标准、目标平台、性能阈值等关键指标未指定
- 澄清时**一次性**把待确认问题集中提出（推荐用 AskUserQuestion 批量提问），避免反复打断用户。
- 用户明确回复后才进入实现阶段；澄清过程中**禁止**提前写业务代码。

### 1.2 任务进行中的不确定点确认（强制弹窗）

任务进行中（编码、调试、修复、重构任何阶段）一旦遇到以下任一情况，**必须立即停止并弹窗向用户确认**，**禁止擅自改逻辑、改需求、改功能**：

- 对需求不理解、对任务不清晰
- 遇到不确定的技术选型、依赖行为、API 契约
- 觉得不对劲、觉得需求有问题 / 缺陷 / 漏洞
- 发现本次任务范围以外的代码、功能、逻辑、需求有问题（即便看似是 bug，也**不擅自改**，先弹窗报告）
- 实现方案与既有约束、project_memory 冲突
- 想推翻用户之前的决策或自己之前给出的方案

### 1.3 弹窗确认的格式要求（强制）

每次弹窗确认任何事项，**必须**：

1. **给出最推荐的方案**——不是罗列选项让用户选，而是明确推荐某一个
2. **说明推荐理由和原因**——技术依据、成本、风险、收益
3. 若要推翻用户之前的决策或自己之前给出的方案，**必须说明推翻理由**，说服用户后才执行
4. 不确定点澄清前，**禁止**继续推进相关代码改动

## 2. 同步输出测试用例与验收标准（TDD 强制）

**触发时机**：需求澄清完成后、正式编码前。**强制 TDD（RED → GREEN → REFACTOR），不得跳过任何阶段。**

### 2.1 验收标准
- 功能完整性：列出本次必须实现的能力点
- 正确性：每个能力点的预期行为
- 边界与异常：空值 / 越权 / 失败路径的处理
- 性能阈值（如适用）：首屏时间、接口 RT、包体积等
- 兼容性：目标平台（Web / 桌面端 / 移动端）与浏览器/系统版本

### 2.2 测试用例
每个用例包含：用例编号 / 场景 / 前置条件 / 操作步骤 / 预期结果 / 优先级。

最少覆盖：
- 主流程正向用例（P0）
- 关键异常路径用例（P0）
- 边界值与权限校验用例（P1）
- UI 交互与状态切换用例（P1，前端任务必填）

清单输出后**同消息**展示给用户，再开始写代码。

### 2.3 TDD 三阶段强制流程

每个新功能/修改必须按以下顺序执行，**禁止先写实现再补测试**：

1. **RED 阶段**：先写测试文件，断言预期行为（断言失败而非导入失败）。运行 `pnpm test <file>` 确认测试因实现缺失而失败。
2. **GREEN 阶段**：写最小实现让测试通过。若 GREEN 阶段发现真实 bug（如 `Number("")===0` 这类边界问题），**立即修复**，并在开发日志记录 TDD 价值。
3. **REFACTOR 阶段**：将纯函数/纯逻辑接入 UI/路由，保持测试全过。运行 `pnpm typecheck` 确认无新类型错误。

### 2.4 自动化测试基础设施

- 单元测试：vitest 2.x（项目根 `vitest.config.ts` + `vitest.setup.ts`）
- 测试环境变量：`vitest.setup.ts` 注入 `DATABASE_URL` / `BETTER_AUTH_SECRET` 等，避免 env.ts 在模块加载时 process.exit(1)
- 测试文件命名：`<module-name>.test.ts`，与被测文件同目录
- 纯函数优先：业务逻辑抽离为纯函数（如 `publish-form.ts` / `profile-form.ts`），便于在 node 环境下测试，避免引入 jsdom / RTL 重型依赖
- 依赖注入：服务层通过 `deps: { db }` 参数注入，便于传 mock 单元测试

## 3. 自测与验收评估

**触发时机**：开发完成、提交代码前。

- 严格按第 2 节的测试用例**逐条**自测，不得跳过。
- 自测方式优先级：
  1. 自动化测试（单元 / 集成 / E2E）
  2. 启动服务后手工调用接口 / 页面操作
  3. 代码审查 + 类型检查（`pnpm typecheck`）+ lint
- 自测结果记录在《测试用例与验收清单》中，标注 ✅ 通过 / ❌ 失败 / ⚠️ 部分通过。
- 任一 P0 用例未通过，**禁止**进入提交环节。
- **回归验证**：每次修复后必须重新执行完整测试套件（`pnpm test`），不仅是失败的用例，避免引入回归。

## 4. 问题自动修复（bug 彻底修复原则）

**触发时机**：自测发现 bug 或偏离验收标准时。

- 发现问题后**立即**分析根因并修复，不得搁置、不得绕过、不得用 try/catch 掩盖。
- **彻底修复原则**：
  - 同一类 bug 必须找到根因（root cause），不能只在调用点打补丁
  - 修复后必须验证同类调用点是否都安全
  - 跨端实现不一致时，必须对齐到正确实现（如 mobile 与 desktop 的 `use-auth` 差异，以正确方为准）
  - 涉及第三方 API 变更（如 expo-router 移除 useColorScheme 导出），必须按官方迁移指南修正
- 修复后必须**重新执行完整自测**（不仅是失败的那一条），避免引入回归。
- 反复迭代直到满足发布标准：
  - 无 bug：所有 P0 用例通过，无控制台 error / 网络请求失败
  - 功能完整实现：需求中列出的能力点全部可用
  - 所有流程正常跑通：端到端主流程能走完
  - 无性能问题：首屏、关键接口 RT、包体积符合验收标准
- 如遇技术性阻塞（无法在当前约束下修复），应停止修复并向用户报告，附带根因分析与可选方案，不得掩盖问题。

## 5. 提交与开发日志

**触发时机**：自测全部通过后。

### 5.1 提交规范
- 每个**独立**功能/修复用一次提交，禁止一次提交混杂多个不相关变更。
- 提交信息格式：
  ```
  <type>(<scope>): <subject>

  <body>
  ```
  type ∈ `feat | fix | refactor | docs | chore | test | perf`；scope 为模块名（如 desktop / api / api-client）。
- 必须先 `git status` + `git diff` 复核变更，禁止 `git add -A`（避免误提交 .env、密钥、大文件）。
- 敏感文件（.env、密钥、credentials）一律禁止提交。

### 5.2 推送到 Gitee + GitHub
- **所有 Lynx 项目代码必须同时推送到 Gitee 和 GitHub**（硬约束，见 project_memory）。
- GitHub remote 名为 `github`，仓库 `https://github.com/woaini737696/Lynx-Kit.git`。
- GitHub 操作需配置代理 `http://127.0.0.1:22307`。
- 推送前确认目标分支正确，避免直接推送到 main/master（除非用户明确要求）。
- 推送命令示例：
  ```powershell
  git push origin master  # Gitee
  $env:HTTPS_PROXY="http://127.0.0.1:22307"; $env:HTTP_PROXY="http://127.0.0.1:22307"; git push github master  # GitHub
  ```

### 5.3 开发日志
每次迭代在 `docs/dev-log/` 目录下追加一份开发日志，命名格式：`YYYY-MM-DD-<scope>-<brief>.md`。

内容至少包含：
- 任务概述（本次做了什么）
- 变更清单（文件级，新增 / 修改 / 删除）
- 测试用例与验收清单（引用第 2 节产出，含自测结果）
- **TDD 合规说明**：RED / GREEN / REFACTOR 三阶段执行情况，TDD 在 GREEN 阶段捕获的真实 bug
- 遗留问题与后续计划（如有）
- 提交哈希与分支名

## 6. 阿里云服务器部署约束（第零章 · 最高优先级）

> ⚠️ 本章为最高优先级硬约束，优先于一切其他章节。任何与之冲突的操作一律禁止。

**触发时机**：任何涉及部署到阿里云服务器的任务。

- **服务器规格**：2 核 2G（资源极其有限，禁止超限使用）。
- **服务器信息**：`47.119.185.135`（root 用户）。
- **强制规则**：必须在本地构建好产物再部署，**绝不允许在服务器上执行任何构建、打包、安装依赖、编译操作**。
  - 允许上传的产物类型：已构建完成的二进制、已打包的 Docker 镜像 tar、已 `pnpm build` 后的 `dist/`、`out/` 静态资源
  - 禁止在服务器执行：`pnpm install`、`pnpm build`、`npm install`、`npm install <pkg>`、`cargo build`、`go build`、`docker build`、`mvn package`、`pip install`、`yarn install` 等任何会拉取依赖或编译的动作
  - 允许在服务器执行：`apt install <系统包>`（如 nginx、postgresql 扩展）、`CREATE EXTENSION`（数据库扩展）、写配置文件、`scp`/`rsync` 上传产物、启动/重启进程
- 部署前必须本地完成完整自测，确认产物可运行后再上传。
- 上传方式优先：本地 `docker build` → `docker save` → `scp` 到服务器 → `docker load` → `docker run`；或直接上传构建好的静态产物。
- 服务器只承担"运行时"职责（运行已构建好的进程、反代、TLS 终止、安装系统级扩展），不承担"构建时"职责。
- 违反此约束导致服务器 OOM / 崩溃的，按严重事故处理。

### 6.1 历史事故警示（真实案例）

**事故**：曾在服务器执行 `npm install tsx` 安装单个包，导致 2C2G 服务器 OOM 宕机，强制重启才恢复。

**根因**：`npm install` 即使安装单个包，也会触发依赖树解析 + 磁盘写入 + 内存占用峰值，2C2G 服务器内存不足以承受。

**教训**：服务器上**任何**形式的 `npm install` / `pnpm install` / `yarn install` 都是禁止的，无论安装的是单个包还是全部依赖。即使只装一个看似轻量的包（如 tsx），也会触发完整的依赖解析流程导致 OOM。

**正确做法**：本地安装 + 本地构建 + 仅上传构建产物到服务器。

## 7. 文件清理约束

**触发时机**：每次任务完成、提交代码前。

- 提交前**必须**清理以下类型文件，禁止入库：
  - 无关文件：与本次任务无关、误生成的文件
  - 临时文件：`*.log`、`*.tmp`、`*.swp`、`*.bak`、`renderer-debug.log`、`electron-stdout.txt`、`electron-stderr.txt`、`COMMIT_MSG.txt` 等提交消息临时文件
  - 无用文件：构建中间产物（`out/`、`dist/`、`dist-electron/`、`.next/`、`build/`、`release/`、`installer/`、`.turbo/`、`coverage/`）
  - 垃圾文件：`.DS_Store`、`Thumbs.db`、`*.tsbuildinfo`、IDE 缓存
  - 重复代码：同一逻辑的多份拷贝、已废弃但未删除的旧实现
- `.gitignore` 必须与上述清理规则保持一致；如发现新类型的临时产物，应同步补充到 `.gitignore`。
- 提交前执行 `git status` 复核，确认无任何上述类型文件被跟踪。
- **禁止使用 `git add -A` 或 `git add .`**，必须按文件名逐个添加，避免误入库。
- 服务器部署目录同样遵守：只保留运行所需的产物，部署完成后立即清理上传过程产生的临时文件。
- **每次任务完成后强制清理**（不浪费磁盘空间）：
  - 删除本次任务过程中产生的所有临时文件、调试产物、中间产物
  - 删除冗余代码、废弃实现、重复拷贝
  - 删除与代码无关的任何内容（文档草稿、调试脚本、临时 zip 包等）
  - 与代码无关的内容**禁止**提交到 Git 和服务器
  - 任务收尾时执行 `git status` 复核，确认工作区干净、无遗留垃圾

## 8. 代码编写原则（强制）

**触发时机**：每次写代码时。

### 8.1 极简原则
- **能少写就不多写**：不添加未被请求的功能、配置、注释、类型注解。
- **能用简单方案就不做复杂**：
  - 不为假设性未来需求做抽象（YAGNI）
  - 三行相似代码优于一个过早抽象
  - 不加无意义的包装层 / 适配器
  - 不加未被请求的 docstring / 注释（代码自解释即可）
- **错误处理最小化**：只在系统边界（用户输入、外部 API）校验；内部代码信任框架保证，不防御性编程。

### 8.2 共用原则
- **能共用的组件/代码就绝不重复写**：
  - 跨端共用逻辑抽到 `packages/shared` 或 `packages/api-client`
  - 表单校验逻辑抽为纯函数（如 `publish-form.ts` / `profile-form.ts`），跨端复用
  - 服务层通过依赖注入（`deps: { db }`）便于测试与复用
  - UI 组件抽到 `packages/ui-web` / `packages/ui-mobile`，不跨端重复实现
- **发现重复立即消除**：提交前 review，发现拷贝粘贴代码必须重构为共用函数。

### 8.3 一致性原则
- 跨端实现必须对齐：mobile / desktop / web 三端对同一后端契约的字段使用必须一致（如 `accessToken` vs `token`）。
- 枚举值使用必须与类型定义对齐：禁止凭记忆写字面值（如 `clarify` vs `clarifying`），必须 import 枚举常量。

## 9. 架构师视角全局架构评审（每次迭代必做）

**触发时机**：每次迭代交付前。

站在架构师角度，从以下 4 个维度评审本次改动与全局架构：

### 9.1 健壮性（Robustness）
- 错误处理是否完整：边界值、空值、越权、失败路径
- 依赖版本是否一致：monorepo 中同一依赖（如 drizzle-orm）是否多版本并存
- 类型安全是否到位：是否有 `any` 滥用、类型断言绕过

### 9.2 扩展性（Extensibility）
- 新增功能是否需要改动多处（开闭原则）
- 抽象层次是否合理：不过度抽象，也不欠缺抽象
- 跨端复用是否到位：能否在 mobile/desktop/web 间复用

### 9.3 快速迭代性（Agility）
- 测试是否快速：单元测试 < 1s，集成测试 < 10s
- 反馈循环是否短：typecheck / lint / test 是否能在秒级完成
- 部署是否顺畅：本地构建 → 上传 → 运行的链路是否打通

### 9.4 性能体验（Performance）
- 首屏时间、关键接口 RT、包体积是否符合验收标准
- **桌面端选型已确定使用 Electron**（决策依据：用户体验、性能、快速迭代便利性 > 体积问题；Electron 生态成熟、React 栈无缝、调试链路短；81MB 在现代网络下可接受）。**禁止再提迁移 Tauri 建议。**
- 桌面安装包体积约束：Electron 自带 Chromium 运行时（~70MB 不可压缩），项目代码仅 ~1.5MB；不再追求压到 30MB 以下
- 移动端包体积、启动时间、内存占用是否达标

评审结果记录在开发日志中，不符合标准的项给出优化建议。

## 10. 迭代交付与下一步建议（强制）

**触发时机**：每次任务完成后。

- 每次交付后，必须给出**详细的下一步迭代建议**，按 P0-P2 优先级分层：
  - **P0（阻塞性）**：影响用户实际使用、安全、数据完整性的问题
  - **P1（工程质量）**：类型安全、测试覆盖、架构债务
  - **P2（体验优化）**：性能、UI 打磨、文档
- 建议需包含：问题描述、影响范围、建议方案、优先级理由。
- 建议记录在开发日志末尾，并在对话中向用户展示。

## 11. 打包交付（桌面安装包）

**触发时机**：用户要求"打包验收"或"安装包"时。

- **桌面端选型**：已确定使用 Electron，**不再考虑迁移 Tauri**（见 §9.4 决策依据）。
- **打包命令**：`pnpm --filter @lynxkit/desktop build`（electron-vite build + electron-builder）。
- **产物路径**：`apps/desktop/release/`（已被 .gitignore 排除，不入库）。每次桌面端有更新都必须重新打包到此目录。
- 打包前必须确认：
  1. 全量测试通过（`pnpm test`）
  2. typecheck 通过（`pnpm typecheck`）
  3. 无遗留临时文件（`git status` 干净）
  4. 开发日志已写入
- 打包后告知用户产物路径与体积，由用户下载验收。

## 12. 附则

- 本规范自项目根目录生效，适用于所有 LynxKit 仓库的子项目（apps/、packages/、scripts/）。
- 与 project_memory.md 中的硬约束冲突时，以 project_memory.md 为准。
- 规范本身可迭代，更新后需同步告知所有协作者。
