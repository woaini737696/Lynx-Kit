# LynxKit 开发流程规范

> 本文档为 LynxKit 项目的强制开发规范。每次接收任务/需求时必须严格遵守，不得跳过任何环节。

## 1. 需求澄清（任务启动前）

**触发时机**：每次接收到新的任务或需求时，**先澄清，再编码**。

- 收到任务后，先通读任务描述与上下文（含 memory、topics、相关文件）。
- 如存在以下任一情况，**必须**通过对话向用户确认，完全澄清后才能开始实现：
  - 需求描述存在歧义或多种合理解读
  - 涉及的技术选型、依赖方向、对外契约不明确
  - 任务范围边界模糊（不知道做不做、改不改）
  - 与既有架构、约束、已沉淀的 project_memory 存在潜在冲突
  - 验收标准、目标平台、性能阈值等关键指标未指定
- 澄清时**一次性**把待确认问题集中提出（推荐用 AskUserQuestion 批量提问），避免反复打断用户。
- 用户明确回复后才进入实现阶段；澄清过程中**禁止**提前写业务代码。

## 2. 同步输出测试用例与验收标准

**触发时机**：需求澄清完成后、正式编码前。

为本次任务输出一份《测试用例与验收清单》，至少包含：

### 2.1 验收标准
- 功能完整性：列出本次必须实现的能力点
- 正确性：每个能力点的预期行为
- 边界与异常：空值 / 越权 / 失败路径的处理
- 性能阈值（如适用）：首屏时间、接口 RT、包体积等
- 兼容性：目标平台（Web / 桌面端 / 移动端）与浏览器/系统版本

### 2.2 测试用例
每个用例包含：用例编号 / 场景 / 前置条件 / 操作步骤 / 预期结果 / 优先级。

最少覆盖：
- 主流程正向用例（P0）
- 关键异常路径用例（P0）
- 边界值与权限校验用例（P1）
- UI 交互与状态切换用例（P1，前端任务必填）

清单输出后**同消息**展示给用户，再开始写代码。

## 3. 自测与验收评估

**触发时机**：开发完成、提交代码前。

- 严格按第 2 节的测试用例**逐条**自测，不得跳过。
- 自测方式优先级：
  1. 自动化测试（单元 / 集成 / E2E）
  2. 启动服务后手工调用接口 / 页面操作
  3. 代码审查 + 类型检查（`pnpm typecheck`）+ lint
- 自测结果记录在《测试用例与验收清单》中，标注 ✅ 通过 / ❌ 失败 / ⚠️ 部分通过。
- 任一 P0 用例未通过，**禁止**进入提交环节。

## 4. 问题自动修复

**触发时机**：自测发现 bug 或偏离验收标准时。

- 发现问题后**立即**分析根因并修复，不得搁置、不得绕过。
- 修复后必须**重新执行完整自测**（不仅是失败的那一条），避免引入回归。
- 反复迭代直到满足发布标准：
  - 无 bug：所有 P0 用例通过，无控制台 error / 网络请求失败
  - 功能完整实现：需求中列出的能力点全部可用
  - 所有流程正常跑通：端到端主流程能走完
  - 无性能问题：首屏、关键接口 RT、包体积符合验收标准
- 如遇技术性阻塞（无法在当前约束下修复），应停止修复并向用户报告，附带根因分析与可选方案，不得掩盖问题。

## 5. 提交与开发日志

**触发时机**：自测全部通过后。

### 5.1 提交规范
- 每个**独立**功能/修复用一次提交，禁止一次提交混杂多个不相关变更。
- 提交信息格式：
  ```
  <type>(<scope>): <subject>

  <body>
  ```
  type ∈ `feat | fix | refactor | docs | chore | test | perf`；scope 为模块名（如 desktop / api / api-client）。
- 必须先 `git status` + `git diff` 复核变更，禁止 `git add -A`（避免误提交 .env、密钥、大文件）。
- 敏感文件（.env、密钥、credentials）一律禁止提交。

### 5.2 推送到 Gitee
- 所有代码必须推送到 Gitee 仓库。
- 推送前确认目标分支正确，避免直接推送到 main/master（除非用户明确要求）。

### 5.3 开发日志
每次迭代在 `docs/dev-log/` 目录下追加一份开发日志，命名格式：`YYYY-MM-DD-<scope>-<brief>.md`。

内容至少包含：
- 任务概述（本次做了什么）
- 变更清单（文件级，新增 / 修改 / 删除）
- 测试用例与验收清单（引用第 2 节产出，含自测结果）
- 遗留问题与后续计划（如有）
- 提交哈希与分支名

## 6. 阿里云服务器部署约束

**触发时机**：任何涉及部署到阿里云服务器的任务。

- **服务器规格**：2 核 2G（资源极其有限，禁止超限使用）。
- **强制规则**：必须在本地构建好产物再部署，**绝不允许在服务器上执行任何构建、打包、安装依赖、编译操作**。
  - 允许上传的产物类型：已构建完成的二进制、已打包的 Docker 镜像 tar、已 `pnpm build` 后的 `dist/`、`out/` 静态资源
  - 禁止在服务器执行：`pnpm install`、`pnpm build`、`npm install`、`cargo build`、`go build`、`docker build`、`mvn package`、`pip install`、`yarn install` 等任何会拉取依赖或编译的动作
- 部署前必须本地完成完整自测，确认产物可运行后再上传。
- 上传方式优先：本地 `docker build` → `docker save` → `scp` 到服务器 → `docker load` → `docker run`；或直接上传构建好的静态产物。
- 服务器只承担"运行时"职责（运行已构建好的进程、反代、TLS 终止），不承担"构建时"职责。
- 违反此约束导致服务器 OOM / 崩溃的，按严重事故处理。

## 7. 文件清理约束

**触发时机**：每次任务完成、提交代码前。

- 提交前**必须**清理以下类型文件，禁止入库：
  - 无关文件：与本次任务无关、误生成的文件
  - 临时文件：`*.log`、`*.tmp`、`*.swp`、`*.bak`、`renderer-debug.log`、`electron-stdout.txt`、`electron-stderr.txt`
  - 无用文件：构建中间产物（`out/`、`dist/`、`dist-electron/`、`.next/`、`build/`、`release/`、`installer/`、`.turbo/`、`coverage/`）
  - 垃圾文件：`.DS_Store`、`Thumbs.db`、`*.tsbuildinfo`、IDE 缓存
  - 重复代码：同一逻辑的多份拷贝、已废弃但未删除的旧实现
- `.gitignore` 必须与上述清理规则保持一致；如发现新类型的临时产物，应同步补充到 `.gitignore`。
- 提交前执行 `git status` 复核，确认无任何上述类型文件被跟踪。
- **禁止使用 `git add -A` 或 `git add .`**，必须按文件名逐个添加，避免误入库。
- 服务器部署目录同样遵守：只保留运行所需的产物，部署完成后立即清理上传过程产生的临时文件。

## 8. 附则

- 本规范自项目根目录生效，适用于所有 LynxKit 仓库的子项目（apps/、packages/、scripts/）。
- 与 project_memory.md 中的硬约束冲突时，以 project_memory.md 为准。
- 规范本身可迭代，更新后需同步告知所有协作者。
